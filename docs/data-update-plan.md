# Schulferienklar Data Update Plan

Schulferienklar depends on reliable holiday and public holiday data. This document defines how data should be added, checked, and published.

## Scope

Schulferienklar currently maintains two main data groups:

- School holiday data in `data/holidays`
- Public holiday data in `data/public-holidays`

The app publishes selected data files under:

- `app/public/data/holidays`
- `app/public/data/public-holidays`

## Current coverage

### School holidays

School holiday data covers all 16 German Bundesländer.

The data may include:

- official school holidays
- Unterrichtsfrei entries
- Bundesland-specific special free days

### Public holidays

Public holiday data covers all 16 German Bundesländer for the years 2026 to 2030.

The data distinguishes between:

- statewide public holidays
- regional public holidays
- local public holidays

Regional and local holidays may be present in the data, but they should not be shown in the default calendar as statewide holidays unless they apply to the entire Bundesland.

## Update principles

### 1. Prefer official sources

Use official state or government sources whenever possible.

Examples:

- Bundesland education ministry pages for school holidays
- official public holiday laws or government pages
- official state portals

Secondary sources may be used only for cross-checking, not as the main source when an official source is available.

### 2. Keep source metadata

Each dataset should keep source information where possible:

- source name
- source URL
- last checked date
- trust level
- notes about regional or local exceptions

### 3. Mark regional and local exceptions clearly

Some public holidays are not statewide.

Examples:

- Mariä Himmelfahrt in parts of Bayern
- Augsburger Friedensfest in Augsburg
- Fronleichnam in parts of Sachsen and Thüringen

These entries should be marked as regional or local and excluded from the default statewide calendar unless they apply to the whole Bundesland.

### 4. Do not guess unpublished future data

For years after 2030, add data only when reliable sources are available.

If a Bundesland has not published school holiday data for a future year, do not infer it from patterns unless the dataset clearly marks the entry as projected or provisional.

The preferred approach is:

- official confirmed data first
- cross-check with secondary sources
- add notes when data is incomplete or provisional

## Adding new public holiday years

Recommended process:

1. Create or update public holiday JSON files under `data/public-holidays`.
2. Copy app-facing files to `app/public/data/public-holidays`.
3. Regenerate the public holiday index.
4. Validate public holiday data.
5. Validate school holiday data.
6. Build the app.

Commands:

```bash
node scripts/generate-public-holidays-index.mjs
cp data/public-holidays/index.json app/public/data/public-holidays/index.json
node scripts/validate-public-holidays.mjs
node scripts/validate-holidays.mjs
cd app
npm run build
cd ..
```

## Adding new school holiday data

Recommended process:

1. Add or update files under `data/holidays`.
2. Copy app-facing files to `app/public/data/holidays`.
3. Regenerate the holiday index.
4. Validate school holiday data.
5. Validate public holiday data.
6. Build the app.

Commands:

```bash
node scripts/generate-holiday-index.mjs
cp data/holidays/index.json app/public/data/holidays/index.json
node scripts/validate-holidays.mjs
node scripts/validate-public-holidays.mjs
cd app
npm run build
cd ..
```

## SEO landing pages

When new years are added, regenerate the SEO landing pages if the app should expose those years to search engines.

Current landing pages are generated by:

```bash
cd app
node scripts/generate-seo-pages.mjs
npm run build
cd ..
```

After regenerating SEO pages:

- verify the generated pages
- verify `sitemap.xml`
- submit or refresh the sitemap in Google Search Console
- submit or refresh the sitemap in Bing Webmaster Tools

## Validation checklist

Before merging data updates:

- [ ] JSON files are valid
- [ ] IDs are unique
- [ ] date formats are valid
- [ ] Bundesland codes are valid
- [ ] regional/local public holidays are marked correctly
- [ ] default calendar inclusion is correct
- [ ] indexes are regenerated
- [ ] app build passes
- [ ] representative pages are manually checked

## Release checklist

Before publishing:

- [ ] Run public holiday validation
- [ ] Run school holiday validation
- [ ] Run app build
- [ ] Check at least one large Bundesland
- [ ] Check at least one city state
- [ ] Check a regional public holiday case
- [ ] Check sitemap after deployment
- [ ] Check Search Console and Bing after deployment

## Notes for future improvements

Possible future improvements:

- automate public holiday generation for additional years
- add per-page source references
- add visible `last checked` information to the app
- add structured data for landing pages
- add iCal subscription feeds instead of only downloadable ICS files
- add a data issue report template
